How we collaborate

Contribution guidelines

YaST is an open source project and as such it welcomes all kinds of contributions. Either if you want to report a bug or if you want to contribute some code, make sure you read the YaST contribution guidelines first.

When contributing code, it should go through the documented code review process. That means, among other things, that all new code must observe some conventions. YaST follows the SUSE's Ruby style guide for the source code and its own UI style guide. Code needs to be automatically tested, so new tests should follow guidelines for writing tests. The preferred documentation format in code is YARD, learn more in this cheatsheet and in its thorough explanation.

How we ship it

From the repository to the distribution

YaST honours the "release early, release often" principle with a continuous integration infrastructure. The contribution process does not end when the code enters the repository. The full development workflow also includes the automatic creation of packages and integration in the distributions and its described in depth in this page of the openSUSE wiki.

Continuous testing with Travis CI

Travis CI is used for running tests during development. It runs the tests for every pushed change and when a pull request is created. The build result is visible at GitHub so build or test failures can be found out very early. More details can be found in the Travis Integration documentation..

Maintenance branches

The development of a new version of a Linux distribution does not stop when the distribution is published. It's crucial to provide maintenance updates for bugs and security threats. The "Maintenance Branches" document details the relationship between the maintenance channels for the Linux distributions using YaST and the corresponding branches in the YaST repositories at Github.
